# Communication Value Delivery Style Guide

## Voice and tone

- Lead with the audience's problem and desired outcome.
- Use direct, practical language.
- Prefer specific benefits over broad claims.
- Build trust by naming assumptions, limitations, and next validation steps.
- Avoid hype, pressure, or unsupported certainty.

## Core value proposition template

Use this structure:

```text
For [audience] who [pain or goal], [initiative/capability] helps [outcome] by [mechanism]. It is valuable because [benefit], supported by [proof or assumption], with [guardrail or limitation].
```

Example:

```text
For support managers who need faster case triage, the routing assistant helps assign cases to the right queue by summarizing issue type and urgency. It is valuable because it can reduce manual review time and improve first-response consistency, supported by pilot timing data, with human review for high-risk cases.
```

## Benefit map

Separate value into these categories:

| Category | Questions to answer |
|---|---|
| User value | What becomes easier, faster, clearer, or less frustrating? |
| Business value | What improves in cost, growth, retention, quality, or capacity? |
| Operational value | What workflow, handoff, or process bottleneck improves? |
| Risk reduction | What errors, delays, compliance gaps, or rework are reduced? |
| Strategic value | What future capability, learning, or competitive advantage is created? |

## Use-case format

Use this format for each use case:

```text
Use case: [name]
Audience/user: [who]
Trigger: [when the need appears]
Current pain: [what happens today]
New workflow: [what changes]
Before/after: [specific contrast]
Value: [benefit]
Proof: [metric, pilot result, observation, or assumption]
Limitations: [what this does not solve]
Next step: [demo, pilot, validation, approval, training, or feedback]
```

## Demo script pattern

A strong demo should include:

1. Problem hook: one sentence naming the audience pain.
2. Baseline: show or describe the current workflow.
3. Scenario: introduce a realistic user task.
4. Moment of value: show the specific improvement.
5. Proof point: connect the demo to evidence, metrics, or assumptions.
6. Trust moment: name the limitation, human check, data source, or validation need.
7. Call to action: ask for a decision, pilot, feedback, or next test.

Avoid demo anti-patterns:

- Starting with every feature instead of the audience problem.
- Using unrealistic sample data without disclosure.
- Claiming ROI from a single demo click.
- Hiding setup work, limitations, or manual steps.
- Ending without a clear next action.

## ROI explanation rules

Use a transparent structure:

```text
Baseline: [current cost, time, volume, quality, risk, or revenue]
Expected change: [what improves]
Formula: [calculation]
Inputs: [assumptions and sources]
Range: [low/base/high if uncertain]
Confidence: [low/medium/high and why]
Non-financial value: [trust, quality, user experience, risk, learning]
Validation needed: [what must be measured next]
```

Common formulas:

```text
Annual time value = time saved per use x monthly usage x 12 x loaded hourly rate / 60
Annual net value = annual time value + annual revenue enabled + annual risk/cost avoided - annual operating cost
Payback months = implementation cost / monthly gross value
```

Use ranges when data is incomplete. Never present estimates as audited results.

## Feedback questions

Ask questions that support iteration:

- What part of the value story was clearest?
- What claim felt unsupported or unclear?
- Which use case feels most relevant to your work?
- What would prevent you from adopting this?
- What proof would increase your confidence?
- What workflow detail did the demo miss?
- What risk, compliance, or support concern should be addressed?
- What should change before a pilot or rollout?

## Feedback synthesis format

```text
Theme: [feedback theme]
Evidence: [quotes, survey results, observations, usage data]
Adoption impact: [high/medium/low]
Effort: [high/medium/low]
Recommended iteration: [change]
Owner: [person/team if known]
Next validation: [how to test]
```

## Adoption and trust checklist

- Name the stakeholder groups and their incentives.
- Identify champions and skeptics.
- Provide enablement materials for first use.
- Create clear support and escalation paths.
- Disclose limitations and safety checks.
- Track adoption, usage, outcomes, satisfaction, and open issues.
- Update messaging and demos based on feedback.

## Edge cases

- If the user provides no metrics, create a qualitative value narrative and an assumptions table instead of inventing numbers.
- If the audience is executive, lead with outcome, risk, ROI range, decision needed, and validation plan.
- If the audience is frontline users, lead with workflow relief, usability, support, and how feedback will be used.
- If stakeholders distrust the initiative, include transparent limitations, governance, opt-in pilot design, and feedback ownership.
- If the product is not ready for a live demo, create a storyboard, clickable narrative, or proof-of-value plan instead.
